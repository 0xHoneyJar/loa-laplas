## 0 · The one-line frame

The repo becomes its description: *Master of ceremonies for the Loa engine — prepares
the quest, the party, and the dungeon, then refuses the ceremony until all three agree.*
**Laplas** is the preparation discipline (tri-manifest module format + ready check);
**poteau** is the enforcement lattice (Claude Code hooks as law, outside the context
window). Together they convert this substrate's enforcement posture from advisory to
deterministic — prompts are gradient; hooks are law; the orchestrator conducts.

## 1 · Problem statement (from evidence, all verified in this repo)

The substrate's enforcement is **advisory by its own admission** — our README states the
SubagentStart hook "logs violations, it does not block" (`hooks/subagent-start/loa-tool-mandate.sh:140`,
re-verified: `exit 0` always; it is also gate #5 in `observatory/contract/hardness-manifest.json`,
declared `prose`). The open-issue catalogue is precisely what advisory enforcement produces:

1. **#7** — the governed path is bypassable and the bypass is cheaper (compose on-ramp
   friction drives agents around Form-C).
2. **#29** — the craft-gate prompt never includes the task: the gate proves *coherence*,
   not *conformance*; wrong-repo work was approved because the reviewer never saw the task.
3. **#30** — the Form C emitter ignores `surface_class`/`review_routing`: a data-validity
   surface mandating a 3-voice council was silently compiled to single-model review.
4. **#31** — mandated grounding reads were ignored 4/4; room packets have no wired
   mandated-reads channel.
5. **#40's class (partial)** — the S1.4 loiter forensics (observatory cycle,
   `cycles/observatory-graduation/decisions.md`): a fully-live agent ground 68 calls
   against a declared ceiling of 2 — `compose-calls-ceiling` is declared, unenforced,
   and rendered HOLLOW in our own instrument.

The comparative study (brief §1) localizes loa's reliability advantage to the opposite
posture: gates run where "the LLM cannot skip [them] because it is not the LLM's decision."

## 4 · Functional requirements

### FR-A · Ready check at the door (laplas, Phase 0)
Wire `laplas-ready.mjs` into compose dispatch: no ceremony arms without a ready receipt
(`.run/poteau/ready.json`) binding quest/party/dungeon hashes. Six cross-validations
(P601 roles⊆party · P602 tools⊆dungeon · P603 council staffable · P604 gates reachable ·
P605 REL compatible · P606 HITL seated). Decompose `compositions/code-implement-and-review.yaml` into the
three manifests as the worked example (chosen because it exercises council routing,
gates, AND a HITL seat — the three surfaces under-testing would miss, IMP-005); P601–P606 negative fixtures join the suite.
Module-format schemas open their hounfour migration PROPOSAL.

### FR-B0 · The contract smoke test (BLOCKER B1/IMP-003 — runs BEFORE FR-B design)
A fixture session proves, on THIS repo's Claude Code version: Stop `{"decision":"block"}`
forces continuation with reason-in-context · PreToolUse exit 2 denies with stderr fed to
the model · `stop_hook_active` loop-guard honored · UserPromptSubmit injection lands.
If any leg fails, the enforcement design degrades to advisory and the cycle HALTS for
re-design — law is not built on an unverified runtime. CI re-runs the smoke test; hook
contract drift = build refusal (extends P302's closure doctrine to the runtime itself).
A-1 is discharged by this FR, not deferred to SDD.

### FR-B · The lattice lands (poteau, Phase 1)
Vendor `poteau/` (reference implementation = executable spec; invariants PT-1..PT-9 must
not regress). `poteau-gen` emits the hooks fragment (generated, checksummed, drift-refusing
— P401); merge per loa's settings.hooks.json pattern. CI runs the demo as a gate.
**Bootstrap runbook (IMP-008, operator-owned)**: the agent NEVER merges its own hook
config. The runbook ships with FR-B: exact merge command · verification (`poteau-gen
--check` against checksums) · rollback (remove fragment, re-verify) — so the
transition is repairable by the operator without agent help.
**Break-glass authorization (B3)**: `POTEAU_BREAK_GLASS=<reason>` is honored ONLY from
session-launch environment (the agent cannot set env mid-session); reason is mandatory;
scope is a single exit; every use lands in `incidents.jsonl` AND surfaces in the run
report (FR-G) — sensed, single-shot, operator-only.
**Grounding note (verified 2026-06-12, macOS)**: reference demo runs 20/21 here — the
single failure is a BSD `wc -l` whitespace-padding bug in the test harness's string
compare (all six P-codes present; `"       6" != "6"`), to be fixed on vendoring. The
21/21 claim is otherwise reproduced.

### FR-C · The dispatcher speaks to the gate (Phase 2 — closes #29, #31)
The compose dispatcher populates `.run/poteau/run-state.json` per stage: `task` (same
literals the work stage receives), `task_ref` (hash), `mandated_reads` (path + literal H1,
extracted mechanically), `review_routing` from composition YAML. The segment-emitter
includes TASK and SCOPE in every gate prompt so the model-side reviewer and the mechanical
gatekeeper check the same contract. Gatekeeper G2 (P201/P202 task-match + in_scope) and
G3 (P203 H1-echo) enforce.
**Legacy compositions (IMP-004)**: missing `task` literals → REFUSE to arm (the
load-bearing field fails closed); missing `review_routing` → defaults to non-council
surface (G4 skipped, logged); missing `mandated_reads` → empty set (G3 passes
vacuously, logged). A migration note in the composition template names all three.

### FR-D · Council runner (Phase 3 — closes #30)
Implement `gate.council.runner`: flatline-style multi-voice invocation, ≥2 distinct
reviewer ids producing signed council receipts into the packet. Until it exists, P301
stands at compile time — single-model compilation of a council surface REQUIRES the
recorded `--allow-single-model` override. Fail-closed beats silent downgrade.
**Override semantics (IMP-002)**: the override is operator-only (compile-time flag,
never settable from inside a run), requires a rationale string, and is recorded into
BOTH the compile receipt and `incidents.jsonl`; the resulting packets carry
`council_waived: <rationale>` — visible at the verify gate and in FR-G telemetry.
**Reviewer identity (B6)**: a reviewer id = provider + model + session nonce;
independence = distinct provider processes (the headless-CLI trio pattern); each
council receipt binds task_ref + packet hash, so receipts cannot be replayed across
tasks. Cryptographic custody of reviewer keys is FR-E scope.

### FR-E · Key ceremony + CAS (Phase 4)
Replace demo's generate-on-first-use key with per-room provisioned ed25519 keys
(versioned, public keys in the run manifest, 0600, operator-held). Wire move-record's
input hashing to the real CAS (legba integration) so receipts become replay-challengeable.

### FR-F · Liveness watchdog (Phase 5 — closes #40's class)
Feed `.run/poteau/moves.jsonl` to asson liveness verdicts (stall/spin/budget); reap and
compact route through the exit gate as checkpoint packets — forced arrival is judged,
never dropped. Block ceilings checkpoint-and-release with an incident (PT-8: never
imprison, never silently release). **Dependency note**: this realizes the §3.3-amendment
trigger taxonomy (liveness | budget) the Observatory cycle specified, and gives
`compose-calls-ceiling` its hook — the hardness manifest flips that entry prose→hook
when, and only when, this lands.

### FR-G · Observability (Phase 6)
Gate-pass rate, refusal-code histogram, break-glass and max-blocks incidents per
construct → GECKO sensing. A gate whose P-code rate climbs is a gate whose contract is
failing.

## 5 · Technical & non-functional

- **Placement doctrine (the sandwich)**: orchestrator conducts (custody, sequencing) ·
  hook lattice is per-move law (deny, record, inject, refuse-to-stop — never initiate) ·
  prompts/skills are gradient (load-bearing for behavior, never for guarantees).
- **Failure postures, correctly placed**: fences fail OPEN with documented rationale
  (prompt-arm, move-record, compact-clew — loa house pattern); custody fails CLOSED
  (gatekeeper P500; tool-gate on its narrow protected-path surface). P302 converts
  runtime fail-open holes into compile-time refusals (the demo's own jq incident).
- **Generated-config discipline**: `poteau.manifest.json` is the only pen; generated
  files refuse drift (P401); checksums in `.run/poteau/checksums.json` (PT-6).
- **Liveness under failure (B4)**: a gatekeeper crash refuses with P500 + fix text —
  it never hangs a session silently; PT-8 block ceilings checkpoint-and-release with an
  incident; break-glass (B3) is the documented escape. Fail-closed never means
  un-escapable: every closed surface has exactly one sensed exit.
- **Run-scoped state (B8)**: poteau state namespaces per ceremony —
  `.run/poteau/<run_id>/` (matching `.run/compose/<run_id>` convention); parallel
  ceremonies cannot race each other's run-state, moves, or receipts. Session-scoped
  hook state binds the run_id it was armed with.
- **Refusals teach**: every deny names what failed, the exact fix, and the why — error
  codes as code/name/what/fix tables (loa E-code convention, P-codes mirror it).
- **Honest limits carried forward (brief §7), extended by review**: bootstrap is the
  orchestrator's (the agent never merges its own hook config); hooks cannot conduct;
  tool-gate is a fence (accepted bypass classes documented), the closed guarantees live
  in gatekeeper + checksum ledger; receipts prove process compliance, not semantic quality.
  **Bypass model stated honestly (B2)**: poteau governs ARMED sessions. A session that
  never arms (work done entirely outside compose) is NOT constrained by hooks — it is
  constrained by the verify gate (no receipts → no `valid_run`) and by #7's incentive
  design (R-3: the governed corridor must be the cheapest). A-2 is therefore not an
  enforcement assumption: unarmed work is detectable-and-unattestable, not impossible.
  **Key custody stated honestly (B5)**: per-room keys on the same OS user are
  fence-grade custody — tool-gate denies key-path reads, the checksum ledger detects
  tampering, but a same-user agent CAN read the filesystem. Hard custody (OS keychain
  or operator-held off-box keys) is FR-E's bar; until then receipts deter and detect,
  they do not prevent forgery by a determined same-user process.
- **House conventions**: hook scripts under the repo's hook layout with in-file posture
  rationale + provenance; audit JSONL in `.run/`; three-tier packet validation
  (`scripts/handoff-validate.sh`) preserved — poteau is the blocking successor to the
  log-only pair, keeping their packet vocabulary.

## 6 · Scope

**MVP (in brief order)**: FR-A (Phase 0) → FR-B (Phase 1) → FR-C (Phase 2) → FR-D (Phase 3).
**Wave 2**: FR-E (Phase 4), FR-F (Phase 5). **Wave 3**: FR-G (Phase 6).
**Explicitly out**: semantic correctness by mechanism — receipts prove process
compliance, never that the work is GOOD; no FR claims otherwise (IMP-012);
replacing the between-session orchestrator (poteau is in-session law
only); hardening tool-gate beyond fence grade; semantic-quality judgment by mechanism
(stays with councils and the operator); hounfour schema RATIFICATION (this cycle only
OPENS the proposal).
