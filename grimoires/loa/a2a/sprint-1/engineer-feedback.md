## Adversarial Analysis

### Concern 1 — `decompose-result.schema.json` invents an `EMPTY` refusal_reason the §0.1 contract does not have (contract drift)
`laplas/schemas/decompose-result.schema.json:11` adds `"EMPTY"` to the `refusal_reason` enum. The §0.1 pinned contract enumerates exactly `SANITIZE_REJECT|DOMAIN_AMBIGUOUS|ROSTER_INVALID|LLM_FAILURE|GOAL_TOO_LARGE|DETECTOR_TIMEOUT` — no `EMPTY`. Worse, §0.1 maps the *empty-items* case to a **serial** (`fallback_reason: LLM_EMPTY`), not a refusal — but `dag-validate.mjs:22` returns `refusal('EMPTY')` for `items.length === 0`. So the impl both (a) adds a refusal value the contract forbids and (b) classifies the zero-item case as refusal where the contract calls it serial-fallback. S1 is defensible because C2 (S3) owns the real `LLM_EMPTY → serial` mapping upstream (dagValidate should never *see* zero items in the happy path), but the schema is now a superset of the pinned union. **Fix**: either remove `"EMPTY"` from the schema enum and have S3 guarantee dagValidate never receives `[]`, or pin a contract amendment in §0.1 explaining `EMPTY` is an internal-only dagValidate outcome (parallel to how `fail` is documented as exit-3-only in the schema `$comment`). Non-blocking for S1; must be reconciled before S3 emits to stdout.

### Concern 2 — `rosterFromParty` cannot load the repo's own canonical good-party fixture (`tier:"external"` → exit 6)
`laplas/lib/roster.mjs:8` hardcodes `TIERS = {haiku, sonnet, opus}` and `:17-19` rejects any other tier as `ROSTER_INVALID`. But `laplas/test/fixtures/party-good.json:7` declares `{ "role": "reviewer-b", ..., "tier": "external" }`. I ran `loadRoster("./laplas/test/fixtures/module-good.json")` → `{"ok":false,"exit":6,"error":"ROSTER_INVALID: unknown tier 'external' for role 'reviewer-b'"}`. The system's own reference "good" party is un-routable by the roster contract. `external` is a real seat kind elsewhere in the substrate (a non-Anthropic council voice). This is a genuine modeling collision: `tier_ceiling` is being validated against the *Anthropic model tier* vocabulary, but `party.members[].tier` in existing manifests also carries `external`. **Fix**: either (a) admit `external` (and any other real party tiers) into `TIERS`, treating it as "no opus clamp" / unclamped, or (b) decouple — only validate `tier` against the tier vocabulary when the member actually seats a routable role, and map `external` to `tier_ceiling: null`. As written, the moment S3 points `loadRoster` at a real module with an external council voice, it exits 6 before the LLM is ever called. Non-blocking for S1 (no test loads a real module — the unit tests build party objects inline with sonnet/opus/haiku only), but this WILL bite S3.

### Concern 3 — duplicate-role members silently drop the second tier_ceiling (first-write-wins, no warning)
`laplas/lib/roster.mjs:20-22`: `if (seen.has(m.role)) continue;` — when two members share a role with different tiers, the **first** member's tier becomes the ceiling and the second is silently discarded. I verified: party `[{role:rev,tier:opus},{role:rev,tier:haiku}]` → `roster.roles = [{id:"rev", tier_ceiling:"opus"}]`. The haiku member vanishes. For a council where reviewer-a is opus and reviewer-b is haiku under one shared role, the ceiling is decided by array order — pure positional luck. The SDD §4.1 says "each role unique" but the code *tolerates* duplicates by dedup-on-key rather than rejecting them, so the "unique" invariant is silently satisfied by dropping data. **Fix**: either reject duplicate roles outright (exit 6, matching §4.1's "each role unique"), or take `max(tier)` deterministically rather than first-seen. Non-blocking but it's a determinism-adjacent footgun: the same party in a different member order yields a different ceiling.

### Challenged assumption — "domain resolves to exactly one registry entry" but there is no registry; any non-delimited string is accepted as a domain
The SDD C3 says resolve `domain_hint` "to exactly one **registry** entry or flag DOMAIN_AMBIGUOUS." `derive-routing.mjs:resolveDomain` (`:11-16`) does NOT consult any registry — it only rejects empty/comma/semicolon/pipe strings and returns everything else verbatim. I confirmed `resolveDomain("smart contracts")` → `"smart contracts"` (spaces are fine), and `resolveDomain("totally-made-up")` resolves successfully. The "registry" check the SDD describes is degraded into "is it a single non-delimited token." An LLM that hallucinates `domain_hint: "frontend stuff"` produces a *resolved* domain that simply isn't in the known set → `decomposition_confidence: 0` → LOW_CONFIDENCE serial. So the failure is safe (serial, not a mis-route), but the assumption "we resolve against a registry" is not what the code does — it resolves against *grammar*, and uses set-membership only for the confidence signal. The report (Known Limitations) half-acknowledges this ("domain registry is minimal"), but the SDD's DOMAIN_AMBIGUOUS-on-unknown-entry intent is not implemented; only multi-value triggers DOMAIN_AMBIGUOUS. This is a deliberate, defensible deferral — but it should be named as an SDD divergence, not a "minimal registry."

### Alternative not considered — `decomposition_confidence` as a hard binary (1|0) rather than a graded/weighted signal
`derive-routing.mjs:42` sets `decomposition_confidence = (domain ∈ known) ? 1 : 0` and `dag-validate.mjs:50` averages these and compares to the floor (0.5 casual / 0.6 default / 0.7 competitive). Because the per-item signal is binary, the "mean confidence" is just *the fraction of items whose domain is known*. With the casual floor 0.5, a DAG needs **more than half** its items in the known-domain set to fan out; with competitive 0.7, it needs >70%. An alternative worth weighing: carry the model's self-confidence as a *tie-breaker within the gated set* (still never as the gate itself — B7 stays intact), or make the per-item signal graded (e.g. exact-match=1, registry-fuzzy=0.7, unknown=0) so a mostly-known DAG with one novel domain isn't dragged under the floor by a single 0. The binary choice is the safest and simplest (correctly B7-preserving), so this is an acceptable S1 decision — but it's the lever behind Concern 4 (the all-serial risk) and should be a conscious, documented choice, not an accident of "1 iff known."

### The "strict confidence → all-serial" risk: **REAL, confirmed empirically, and currently a silent footgun**
This is the most important finding. I built an un-migrated composition — a dungeon whose rooms declare no `covers_domains` and a party whose members declare no `domain` (i.e. the shape of *every existing manifest in the repo today*; `dungeon-vault.json` and `dungeon-casual.json` fixtures have bare rooms with no `covers_domains`, and `party-good.json` members have no `domain` field). Result:

```
deriveRouting([{a:code},{b:design←a}], unmigrated_ctx)
  → both items decomposition_confidence: 0   (known-domain set is EMPTY: coverage ∅ ∪ role-domains ∅)
dagValidate(...) → { type:"serial", fallback_reason:"LOW_CONFIDENCE" }
```

So an un-migrated composition routes **everything to serial fallback — no fan-out, ever** — and does so *silently* (a clean `serial`, exit 0, no warning that the cause is "you never declared covers_domains or per-role domain"). The known-domain set is `coverage ∪ {role.domain}` (`derive-routing.mjs:48-52`), and **neither input is populated in any existing manifest**: no fixture sets `covers_domains`, and `rosterFromParty` always maps `domain: m.domain ?? null` (`roster.mjs:22`) because no party member declares one. Net: the entire decomposer ships **defaulting OFF** — the headline feature (parallel fan-out, G-1/G-3) is unreachable until operators migrate manifests, and there is no signal telling them why their goals always run serial.

Is that acceptable degradation or a footgun? Both. **Safe**: serial is the correct conservative fallback, B7 is preserved, no mis-route. **Footgun**: the degradation is *invisible*. An operator dogfooding `/compose <goal>` on today's manifests will conclude "the decomposer doesn't fan out" and never learn it's a one-line manifest declaration away. Recommendation (own it in S3, not a blocker for S1): when a `serial/LOW_CONFIDENCE` result is produced **because the known-domain set was empty** (as opposed to genuinely-low per-item confidence), emit a distinct telemetry signal / `fallback_reason` nuance (e.g. `LOW_CONFIDENCE` with detail `"no covered domains declared — composition not migrated"`) so the cause is legible. Otherwise the migration test the SDD C5 promises ("`code-implement-and-review` declares its gate's `covers_domains`") is the *only* thing standing between this feature and "looks broken on every un-migrated comp."
