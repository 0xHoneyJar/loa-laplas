# construct-handoff v1.2 — construct-DDD anti-corruption layer at handoff

> Status: proposal · 2026-05-17 · additive over v1.1, fully backwards-compatible
> Adds two fields closing the construct-domain-boundary translation gap. Composes with operator OS v3.3 Construct Domain Boundaries section and `~/vault/wiki/concepts/construct-domain-boundaries.md` doctrine.

## The gap v1.1 left open

v1.1 added `source_provenance[]` with `source_type` / `privacy_scope` / `use_label` / `decay_class` — these track WHERE a source came from and WHAT authority it carries. They do NOT track WHICH CONSTRUCT's bounded context the source belongs to.

The operator's stated pain (2026-05-17):
> *"absolute clarity with which constructs are being called, what they have access to, what they're reading, the words that actually apply in their domain... the handoff between these agents and the domain constraints that allow for actual translation... what might mean something to one agent in their frame of mind will mean something completely different in another state of mind."*

This is the DDD **translation problem** at construct handoff. Same word — "weight," "scope," "blast radius" — means different things in different constructs. v1.2 closes the gap with the **anti-corruption layer** at the envelope level.

## The two additive fields

### `provenanceEntry.source_construct_affinity` (recommended tier)

Array of construct slugs naming which construct(s)' bounded context the source belongs to.

```json
{
  "source_type": "vault",
  "ref": "wiki/concepts/freeside-as-layered-station.md",
  "privacy_scope": "actor_private",
  "use_label": "background_only",
  "source_construct_affinity": ["the-arcade", "freeside", "observer", "cross-domain"]
}
```

Mirrors the `construct_affinity` field in vault frontmatter (operator's vault, Phase 1 backfill complete 2026-05-17). Lets the receiving construct route / refuse / translate appropriately.

Slugs are validated against `^[a-z][a-z0-9-]*$` (lowercase kebab) — matches installed construct slugs from `construct.yaml` files. Cross-construct doctrine declares `["cross-domain"]` or multi-construct arrays.

### `translation_note` (recommended tier, envelope root)

The anti-corruption layer when an envelope crosses a construct boundary (calling construct's room hands off to a different construct's room).

```json
{
  "translation_note": {
    "from_construct": "k-hole",
    "to_construct": "artisan",
    "note": "Research surfaced 'weight' as confidence-divergence signal. Artisan-domain 'weight' means visual mass + rhythm. Treat the finding as background context for FEEL judgments — NOT as a substitute for material/rhythm scoring.",
    "key_translations": [
      {
        "term": "weight",
        "from_meaning": "confidence-divergence between stated rationale and behavior",
        "to_meaning": "visual mass + rhythmic emphasis in a layout"
      }
    ]
  }
}
```

- `note` (required when translation_note is present) — free-form translation, minLength 30 (rejects empty placeholders).
- `from_construct` / `to_construct` — optional, lowercase kebab construct slugs.
- `key_translations` — optional structured term-by-term mapping. Up to 8 entries.

## What schema enforces vs adapter policy

### Schema-enforced (validator rejects on violation)

- `provenanceEntry.source_construct_affinity[]` items MUST match `^[a-z][a-z0-9-]*$` — lowercase kebab. Rejects `"The-Arcade"`, `"Some Bad Slug"`. See `invalid-invalid-affinity-slug.json`.
- `translation_note.required: [note]` — empty `{}` fails. See `invalid-empty-translation-note.json`.
- `translation_note.note` has `minLength: 30` — vague entries like `"short note"` fail. See `invalid-vague-translation-note.json`.
- `translation_note.key_translations[].required: [term, from_meaning, to_meaning]` — each table row must be complete.

### Adapter-template enforced (policy, not schema)

- When `output_type === "Verdict"` AND the calling construct differs from the consuming construct's expected domain, `translation_note` SHOULD be populated.
- When `source_provenance[].source_type === "vault"`, `source_construct_affinity` SHOULD be populated (since vault pages declare their affinity in frontmatter; the envelope just propagates).
- The adapter template prompts the construct to consider translation when crossing domains, but schema doesn't reject envelopes without it (would over-trigger on intra-construct handoffs).

## Remaining honest limits

Two findings could not be fully schema-enforced and are documented as caller responsibility:

1. **Slug existence in active construct registry** (Flatline-SKP-002 HIGH 760). JSON Schema 2020-12 cannot natively validate against an external registry. Build-time check or external linter could close this; deferred to future cycle.
2. **Cross-field comparison `from_construct != to_construct`** (no-self-translation). Same JSON Schema limit. Caller responsibility; future schema dialect (e.g., JSON Schema with cross-field comparison extensions) could close this.

Both are documented explicitly in schema field descriptions per BB-F6 pattern (honest about what schema enforces vs what adapter / caller enforces).
