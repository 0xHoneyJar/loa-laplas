# construct-handoff v1.1 — vault-aware envelopes + kaironic timing + pushback affordance

> Status: proposal · 2026-05-15 · additive over v1.0, fully backwards-compatible
> Adds three fields to `construct-handoff.schema.json` to close observability gaps named in upstream operator-OS work.

## The three gaps v1.0 left open

The existing handoff schema enforces **what the construct did** (verdict, output_type, gates) but not **what the construct used**, **when it was invoked**, or **what it doubts about its own output**. v1.0 catches confabulation at the rationale layer (per README §"The thesis: observability") but leaves three propagation gaps:

### Gap 1 — privacy + use-label provenance does not flow

When room A consults operator-private context (e.g. `~/vault/`) and emits an envelope to room B, room B has **no signal** that the cited material was operator-private with `use_label: background_only`. Room B can inline the citation into a cross-stakeholder surface and leak operator-private content. The v1.0 schema's `input_refs` and `evidence` are agnostic to privacy class and authority level.

### Gap 2 — kaironic vs chronos invocation timing is invisible

The substrate cannot distinguish "operator invoked this in flow" from "this fired on a cron." Same envelope shape, different operator-experience cost. The operator needs to know which autonomous-cycle envelopes broke flow vs which were quietly composed.

### Gap 3 — operator pushback signal is buried in the verdict

Per the README's invariant 2 (stated reasoning is suspect), envelopes need cross-validation. But the verdict surface (free-form `verdict: object`) doesn't carry a structured **what to push back on** signal. The operator must read the full verdict to surface uncertainty, which collapses at scale: N envelopes = N reads. With a structured `pushback_invitation`, N envelopes = N quick scans.

## The three additive fields

### `source_provenance` (recommended tier)

Array of provenance entries. Each entry declares `source_type`, `ref`, `privacy_scope`, `use_label`, and optional `operator_validated`, `decay_class`, `hash`.

Use cases:
- Vault-sourced material: `source_type: "vault"` + `privacy_scope: "actor_private"` + `use_label: "background_only"`. Downstream rooms inherit the constraint.
- Activated doctrine: `source_type: "activated_doctrine"` + `use_label: "usable"`. Operator promoted it; room may use as authority.
- Agent inference: `source_type: "agent_inference"` + `use_label: "mark_as_contested"`. No external source; agent-derived. Reviewer should weight accordingly.

Composition target: Eileen's loa-straylight recall-wedge schema-contract (`Assertion` + `SignatureEnvelope` + `RecallReceipt`). The `provenanceEntry` shape mirrors what a wedge `RecallReceipt` would carry without adopting the full wedge audit chain. Drop-in compatible when straylight's Phase 24C scaffold settles.

### `kaironic_context` (optional tier)

Object with `trigger` (kaironic / chronos / operator_invoked / upstream_handoff), `flow_state`, `attention_cost`, `interrupt_authorized`.

Use cases:
- Autonomous cycle envelopes carry `trigger: "chronos"` + `interrupt_authorized: false`. Downstream consumer (operator-facing surface) batches for later review instead of interrupting.
- Interactive composition envelopes carry `trigger: "kaironic"` + `interrupt_authorized: true`. Surface promptly.
- Observability layer can query: "show all envelopes with `trigger=chronos` and `attention_cost=high` from the last 7 days" to catch noisy scheduled-cycles that broke flow.

Composes with operator OS v3.2 "Kaironic Time Protocol" (kaironic vs chronos vs operator-paced invocation modes). The envelope makes that protocol's posture inspectable.

### `pushback_invitation` (recommended tier)

Object with `primary_uncertainty` (≤280 chars one sentence), `operator_check` (single specific question), `alternative_paths` (≤4), `confidence` (0..1).

Use cases:
- Pre-digest operator review: instead of reading N verdicts, operator scans N `operator_check` questions and picks the 1-2 worth investigating.
- Cross-validation signal: the construct names its own weakest link. If the named weak link is "no uncertainty," that's itself a signal (probably wrong — agents are rarely certain).
- Composes with operator OS MAY-LATITUDE-5 "Pushback Affordance" rule: agents SHOULD lead with what to doubt, not what to agree with.

Anti-pattern: `primary_uncertainty: "I might be wrong about something"` — vague. Reject in validator (future enhancement) or in adapter prompt.
Pattern: `primary_uncertainty: "The weight-rhythm coupling judgment relied on a vault doctrine loaded as background_only; the verdict assumes 3-plane lens applies to visual-surface, which may not hold."` — specific weak link named.

## Validator behavior — what schema enforces vs what policy enforces

Honest split (corrected in v1.1.1 after Flatline + Bridgebuilder reviews caught misleading prose):

### Schema-enforced (validator rejects on violation)

- **`provenanceEntry` allOf #1**: `if source_type === "vault" then privacy_scope === "actor_private"`. Closes the vault leak vector — a `{source_type: "vault", privacy_scope: "public"}` entry now fails validation. See `tests/fixtures/handoff-packets/invalid-vault-public-scope.json`.
- **`provenanceEntry` allOf #2**: `if source_type === "vault" then use_label !== "usable"`. Raw vault content can never be authority in the same envelope — operator must promote via `source_type: "activated_doctrine"` (post-activation receipt). Closes the authority contradiction. See `invalid-vault-usable.json`.
- **`pushbackInvitation.required`**: `[primary_uncertainty, operator_check]` with `minLength: 20` on each. Rejects empty `{}` and vague entries like "I might be wrong." See `invalid-empty-pushback.json`.
- **`kaironicContext.required`**: `[trigger]`. Rejects empty `{}`. See `invalid-empty-kaironic.json`.

### Adapter-template enforced (policy layer, not schema)

The schema does NOT enforce these — they are policies for adapter-template prompts, validator integration tests, or hook-level checks:

- `output_type: "Verdict"` SHOULD populate `pushback_invitation` (high blast radius outputs benefit most from operator-pushback affordance).
- A construct that consulted vault content SHOULD populate `source_provenance` (no schema-level coupling between `input_refs` content and `source_provenance` presence; adapter prompt + hook-level scan close this gap).
- `kaironic_context.interrupt_authorized` is ADVISORY only — the orchestrator owns the final interruption decision (per BB-F11 review).

### Schema-version coupling

`schema_version` defaults to `"1.0"` but adapters SHOULD emit `"1.1"` when any v1.1 field is present, so v1.0-pinned consumers can refuse cleanly. The schema does NOT couple field presence to version (a v1.1 envelope with `schema_version: "1.0"` validates — this is intentional backwards-compat headroom).

## Schema versioning

Per existing v1.0 contract: `schema_version: "1.1"` for envelopes using the new fields. Validator accepts both 1.0 and 1.1 packets (pattern `^1\.[0-9]+(?:\.[0-9]+)?$`). v1.1.1 patches don't bump the version number — they're additive constraints on existing fields, not new fields.

## Why this lands as PR not as a future cycle

The substrate is 4 days old, three sprints in. The additive surface is small (one schema file + fixtures + one docs page). The composition target (Eileen's straylight recall-wedge) is **also** in active development (Phase 24C scaffolded May 15). Landing the fields now lets the substrate's downstream consumers start populating them as they're built, instead of retrofitting later.

If the field shapes need refinement, that's the value of getting them in front of Eileen + maintainers. The proposal owner welcomes pushback on naming, enum values, and field placement.
